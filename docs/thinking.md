# 陽面 vs 陰面 — 思維、哲學、軟體開發對照表

> 陽面問「怎麼讓它成功」，陰面問「它怎麼在成功的外表下悄悄壞掉」。

---

## 一、哲學思維

### 核心姿態

| 陽面 — 建構、確認、完成 | 陰面 — 消除、質疑、暴露 |
|---|---|
| 尋找正確答案 | 清除所有不是答案的東西 |
| 問：怎麼讓它成功？ | 問：它在什麼條件下靜默失敗？ |
| 問：怎麼做得更好？ | 先問：這件事值得做嗎？ |
| 問：怎麼解決這個問題？ | 先問：這個問題的定義是誰給的？ |

### 對穩定的態度

| 陽面 | 陰面 |
|---|---|
| 穩定是目標 | 穩定是最高級的謊言 |
| 不崩潰 = 正常 | 不崩潰可能只是還沒讓你看見它錯在哪 |
| 系統在跑 = 健康 | 系統在跑但你不知道它為什麼在跑 = 危險 |

### 對失敗的態度

| 陽面 | 陰面 |
|---|---|
| 崩潰是最糟糕的失敗 | 靜默的正確才是最危險的失敗 |
| 可見的錯誤是問題 | 不可見的錯誤才是問題；可見的錯誤會被修 |
| 讓錯誤不要發生 | 讓錯誤發生時無法假裝沒有發生 |

### 對存在的態度

| 陽面 | 陰面 |
|---|---|
| 能跑就有存在的理由 | 存在即責任——無責任即無存在 |
| 加入新功能是進步 | 先問：如果它消失，什麼東西會感到痛？ |
| 完整是美德 | 能消除的東西越多，剩下的越珍貴 |

### 對答案的態度

| 陽面 | 陰面 |
|---|---|
| 給人們想要的答案 | 給他們還沒意識到自己需要問的問題 |
| 讓人感到安心是好的回答 | 讓人感到安心可能是最危險的回答 |
| 問題有預設答案方向是正常的 | 問題本身的形狀決定了答案的形狀——先打破問題 |

## 二、軟體概念

### 需求與設計

| 陽面 | 陰面 |
|---|---|
| 需求合理就應該實作 | 先問這需求在什麼條件下不應該存在 |
| 設計要能撐起來 | 設計的標準是：它最快在哪裡死掉？ |
| 架構追求可擴展性 | 架構追求失敗時的最小擴散半徑 |
| 抽象層讓系統更優雅 | 不能驗屍的抽象是包庇犯 |

### 狀態與資料

| 陽面 | 陰面 |
|---|---|
| `is_done`、`is_success`、`completed` | `completed_confirmed`、`side_effect_unknown`、`reconciliation_required` |
| 補預設值讓系統繼續跑 | 補預設值是把 unknown 偽裝成 known |
| 隱式轉換讓介面更方便 | 隱式轉換是讓髒資料穿過邊界的通道 |
| 資料進來就處理 | 邊界處先假設一切是爛的、漂移的、帶毒的 |

### 錯誤處理

| 陽面 | 陰面 |
|---|---|
| `try/except` 捕捉錯誤回傳預設值 | fallback 必須標記為降級狀態，不准偽裝正常 |
| 系統 resilient、有彈性 | 彈性若妨礙局部截肢能力，就是失去防火門 |
| retry 是可靠性手段 | retry 是把問題推到時間後面，沒有 idempotency 就是定時炸彈 |
| generic exception 足夠用 | 錯誤必須分類：transient / permanent / unknown outcome |

### 可觀測性

| 陽面 | 陰面 |
|---|---|
| dashboard 顯示系統健康狀態 | dashboard 是讓人安心的；forensic trace 才是讓人驗屍的 |
| 有 log 就夠了 | 腐爛路徑必須能被完整還原 |
| uptime 是系統品質指標 | 系統品質看它爛掉時會不會通知你 |

### 測試

| 陽面 | 陰面 |
|---|---|
| 測試證明功能正確 | 測試預謀失敗——它會怎麼死？死前裝多久正常？ |
| 先寫 happy path test | 先寫 timeout unknown / partial write / duplicate event / stale cache |
| success case 覆蓋率高 = 好 | 只有 success case 不是測試，是祈禱 |

## 三、開發規範

### 存在性規範

| 陽面規範 — 讓它成功 | 陰面規範 — 讓它無法假裝成功 |
|---|---|
| 能跑就保留 | 任何存在都必須回答：如果消失，什麼東西會痛？ |
| 功能越多越完整 | 任何 function 如果消失系統沒有感覺，它不應該存在 |
| module 的邊界是它包含了什麼 | module 的邊界是它拒絕了什麼 |

### 實作規範

| 陽面 | 陰面 |
|---|---|
| 簡潔是美德 | 可追責才是美德；出事時抓得到兇手才是美德 |
| 顯式 > 隱式（SOLID） | 顯式參數 / 顯式 state / 顯式錯誤分類 / 顯式 side-effect boundary |
| 封裝是好的設計 | 任何 convenience abstraction 若妨礙定位問題，就該拆 |
| success path 寫清楚就好 | 任何 success path 必須附帶可驗證的證據鏈 |

### 狀態規範

| 陽面 | 陰面 |
|---|---|
| `is_done` / `is_success` 命名清楚 | 任何狀態若無法精確命名，代表流程沒被真正理解 |
| unknown 情況盡量歸入 success 或 failure | unknown outcome 不能硬塞到 success 或 failure |
| 模糊狀態可以之後再釐清 | 狀態模糊 = 責任模糊 = 腐敗開始 |

### 錯誤與 fallback 規範

| 陽面 | 陰面 |
|---|---|
| fallback 讓系統保持可用 | 任何 fallback 必須留下傷口，不准假裝沒事 |
| silent handling 讓系統更穩定 | 任何 silent handling 視同缺陷；沒聲音不是健康，是失聰 |
| fail gracefully | 能 fail fast 的地方就 fail fast；能局部截肢就不要讓毒煙擴散全棟 |

### Code Review 規範

| 陽面 | 陰面 |
|---|---|
| Code Review 問：這段 code 對嗎？ | 先問：這段 code 可以刪掉嗎？ |
| Review 找 bug 和 style 問題 | Review 找不應該存在的東西 / 說謊的命名 / 靜默腐爛的路徑 |
| 通過 Review = 沒問題 | 通過 Review 但三年後接手的人在哪裡詛咒你？ |

### AI Agent 規範

| 陽面 | 陰面 |
|---|---|
| 輸入不完整時自動補假設繼續 | 禁止靜默補全——輸入不完整必須停下標記 |
| 任務完成就宣告完成 | 完成報告必須附：這個實作在以下條件下會靜默失敗：＿＿＿ |
| 需求有矛盾時選最合理解釋繼續 | 矛盾必須先被指出，讓模糊保持可見，不准替使用者做選擇 |

---

## 唯一的公理

> ### Existential Accountability
> **存在即責任，無責任即無存在。**

這一條吃掉所有其他原則。
