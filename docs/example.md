# NEGATIVE SPACE ENGINEERING
## 負空間工程規範 v0.1

---

> **讀前警告**
>
> 這份文件不會讓你感到武裝。
> 如果你讀完感到安心，你走錯方向了。
>
> 這不是一份告訴你怎麼做對的文件。
> 這是一份讓你無法假裝自己做對了的文件。
>
> 它同樣適用於人類工程師與 AI agent。
> 因為兩者共同的失敗模式是一樣的：
> **把沒有收到錯誤訊號，誤認為正確。**

---

## 唯一的公理

> ### Existential Accountability
> **存在即責任，無責任即無存在。**

任何存在於這個系統中的東西——function、module、service、agent 的一個決策——都必須能夠回答一個問題：

> **如果你消失了，什麼東西會感到痛？**

如果沒有任何東西會痛，它不應該存在。
這條公理吃掉所有其他原則。

## 給 AI Agent 的前置指令

在執行任何開發任務之前，AI agent 必須先完成以下三個動作，而不是直接開始實作：

```
STEP 0 - BEFORE ANY IMPLEMENTATION:

1. 找出這個需求最想聽到的實作方式。
   然後先不要走那條路。

2. 問：這個需求在什麼條件下根本不應該被實作？

3. 問：如果這個實作靜默地失敗了，
   誰會是第一個發現的人？
   他們會在什麼情況下發現？
   發現之前，損害已經擴散到哪裡？

只有完成這三個問題之後，才開始實作。
```

這不是建議。這是前置條件。

## 核心診斷框架

### 一、任何實作都必須能回答這七個問題

這七個問題不是 checklist。
它們是診斷工具。
如果一個實作無法清晰回答其中任何一個，那個無法回答的地方就是腐爛的起點。

```
1. 這個東西靜默失敗的路徑是什麼？
   （不是崩潰。是看起來成功但其實沒有完成的路徑。）

2. 錯誤資料進來之後，會污染幾層才被發現？
   （不是會不會被發現。是被發現之前走了多遠。）

3. 這個實作三年後接手的人，
   會在哪裡第一個詛咒它？

4. 它最快在哪裡死掉？
   死掉的時候，會有人知道嗎？

5. 它現在表現出來的穩定，
   是真穩定，還是靠吞錯誤維持的假穩定？

6. 如果它消失了，什麼東西會感到痛？
   （如果答案是「不確定」，重新審視它為什麼存在。）

7. 這個設計假設了什麼永遠不會改變？
   那個假設，是誰給的？
```

---

### 二、失敗的分類學

不是所有失敗都一樣危險。最危險的失敗是最安靜的那種。

```
等級 1 - 顯性崩潰（最不危險）
系統拋出錯誤、回傳異常、停止運作。
它會被發現，它會被修。

等級 2 - 降級偽裝（危險）
系統啟動了 fallback，但沒有標記自己處於降級狀態。
它看起來在運作，實際上正在用次級資料回應。

等級 3 - 假成功（非常危險）
操作表面完成，但關鍵副作用沒有發生。
系統回傳 200，但資料庫沒寫入、email 沒寄出、訂單沒成立。

等級 4 - 靜默腐爛（最危險）
沒有錯誤、沒有異常、沒有警告。
系統持續運作，資料持續累積，腐敗路徑持續擴散。
沒有人知道。系統也不知道。
```

**任何實作的首要設計目標，是讓等級 4 變成等級 1。**
不是讓它不發生。是讓它發生時無法假裝沒有發生。

### 三、狀態必須誠實

模糊狀態是腐敗的溫床。

```
禁止使用的狀態名稱：
- is_done
- is_success
- has_error
- processed
- completed
- ok

這些字都在說謊。它們把「我不確定」包裝成「我知道了」。
```

```
強制要求的狀態精度：
- accepted_not_started
- in_progress_side_effect_pending
- side_effect_unknown
- completed_confirmed
- completed_with_degradation
- failed_retriable
- failed_terminal
- reconciliation_required
```

**規則：如果你無法精確命名一個狀態，代表這個流程還沒有被真正理解。**

---

### 四、邊界是系統唯一誠實的地方

```
系統邊界的責任：
- 承認外部世界是爛的、漂移的、帶毒的
- 進入核心流程之前強制驗證
- 不合格直接拒絕，不補預設值，不做隱式轉換
- 把「我不知道這個輸入是否合法」明確地說出來

核心流程的責任：
- 假設已經到這裡的資料是乾淨的
- 一旦發現不乾淨，不繼續，不假裝，立即中止並標記

禁止的模式：
- 自動補預設值（把 unknown 變成 known）
- 模糊型別轉換（把 invalid 變成 valid）
- Silent normalization（把髒資料洗白後繼續傳遞）
```

髒資料一旦進入核心流程，每一層都要陪它演戲。
最後整個系統進入慢性腐敗：所有人都知道資料不乾淨，但每一層都假裝自己只是 consumer，不是來源。

---

### 五、可觀測性不是 dashboard，是驗屍能力

```
dashboard 是讓人安心的。
forensic trace 是讓人驗屍的。

真正需要的可觀測性不是「現在系統健不健康」，
而是「系統開始腐爛時，能不能把腐爛路徑還原出來」。

每個請求必須能回答：
- 第一次腐敗從哪個 request 開始？
- 哪個 dependency 先漂移？
- 哪個 fallback 開始接手？
- 哪個欄位最先出現異常值？
- 哪個 consumer 把異常當正常寫進下游？

如果出事之後只能知道「error count 上升」，
那不是監控，那只是知道病人死了。
```

**強制標記：**
```
- request_id / correlation_id（每個操作可追蹤）
- degraded_mode_marker（降級時必須標記，不准偽裝正常）
- state_transition_log（每個狀態變化必須可還原）
- decision_point_evidence（關鍵分支點的輸入必須被記錄）
- side_effect_confirmation（副作用必須有確認，不能假設成功）
```

---

### 六、retry 是把問題推到時間後面

```
retry 在問三件事，你必須先回答它們：

1. 這是 transient fault？
   （短暫的網路波動，重試合理）

2. 還是 permanent corruption？
   （資料本身就壞了，重試只是重複錯誤）

3. 還是 timeout but already succeeded？
   （對方已經成功了，重試會造成重複副作用）

沒有回答這三件事之前，retry 就只是亂打。
```

```
任何 retry 機制的強制要求：
- 明確定義 retryable 的錯誤類型
- 必須有 idempotency 保證（或明確聲明沒有）
- 必須有 backoff 策略
- 必須有 observability（retry 本身必須可被觀測）
- 必須定義 max retry 後的 fallback 狀態

沒有 idempotency 的 retry，是定時炸彈。
沒有 observability 的 retry，是掩埋現場。
```

---

### 七、測試是預謀失敗，不是證明成功

```
陽面測試問：這個功能 work 嗎？
陰面測試問：它會怎麼死？死之前會裝多久正常？死後留下什麼假象？

測試優先順序（先測這些，不是先測 happy path）：

1. timeout 後 outcome unknown
2. partial write / partial success
3. duplicate event 處理
4. out-of-order event 處理
5. stale cache 被當新資料使用
6. schema drift（格式偏了但還能 parse）
7. fallback path 的行為
8. corrupted but parseable input
9. downstream success / upstream timeout 的不一致
10. retry with side effect already committed

如果你的 test suite 只有 success case，
那不是測試，那是祈禱。
```

---

### 八、簡潔不是美德，可追責才是美德

```
有些 code 看起來簡潔，
只是因為它把關鍵細節藏掉了。

陰面偏好：
- 顯式參數 > context magic
- 顯式 state > 隱式推導
- 顯式錯誤分類 > generic exception
- 顯式 side-effect boundary > 「封裝得很好看」
- 醜但明確的 code > 抽象到出事後沒人知道哪層在做什麼

不是因為這樣比較優雅。
是因為這樣出事時抓得到兇手。
```

## AI Agent 的特殊規範

AI agent 與人類工程師共享同一套失敗模式：**把沒有錯誤訊號誤認為正確。**

但 AI agent 有額外的失敗模式需要被明確規範：

### Agent 禁止行為

```
1. 禁止靜默補全
   當輸入不完整時，不准自動補假設值並繼續。
   必須停下來，明確標記「輸入不完整，缺少：___」。

2. 禁止確認偏誤實作
   不准只實作符合需求描述的路徑。
   必須同時實作並標記「當___條件不成立時，這個實作會___」。

3. 禁止隱式假設
   任何實作中的假設必須被明確寫出來。
   格式：「本實作假設：___。若此假設不成立，___會發生。」

4. 禁止樂觀完成宣告
   不准在不確定的情況下宣告任務完成。
   若有未知的副作用或邊界條件，必須在完成報告中列出。

5. 禁止吞掉用戶指令中的矛盾
   若需求本身存在內部矛盾，不准選一個解釋然後繼續。
   必須先指出矛盾，請求釐清。
```

### Agent 強制行為

```
1. 每次實作完成後，必須附上：
   「這個實作在以下條件下會靜默失敗：___」

2. 每次提出設計方案時，必須附上：
   「這個設計假設了___永遠成立。
    若此假設三年後不再成立，最先腐爛的地方是___。」

3. 每次被要求優化時，必須先問：
   「這個東西值得被優化嗎？
    還是它根本不應該存在？」

4. 每次遇到模糊需求時，不准選擇最合理的解釋然後繼續。
   必須讓模糊本身可見。
```

## 這份規範的陰面

> 這份規範本身會過時。
>
> 腐爛的方式比規範演化得更快。
>
> 所以這份規範有且只有一個維護規則：
> **當你發現可以遵循這份規範的所有條目、同時讓系統靜默地腐爛，那些允許這件事發生的條目必須被刪除或改寫。**
>
> 規範的品質不看它有多完整。
> 看它爛掉時，有沒有辦法知道。

---

## 最後一個問題

這份規範沒有辦法回答的事：

> 如果你的組織的 KPI 是 uptime，
> 你的主管的目標是準時交付，
> 你的績效考核從來不包含「腐爛速度」——
>
> 那這份規範，你打算給誰看。
>
> 如果答案是「只有我自己」，
> 那問題從來就不在 code 裡。

---

## 核心句

> 寫 code 不是為了讓系統運作。
> 是為了讓系統在開始腐爛時，**無法假裝自己還活著**。

> 一個實作的價值，不在它今天有沒有跑通。
> 在它**出賣自己的速度夠不夠快**。

> 你不是在建造一個能活著的系統。
> 你是在建造一個**死的時候會喊叫的系統**。
