# Overview: samsara-auto-mode

## Goal

Allow Samsara to run in `auto` mode by replacing human gate decisions with a reusable principle-level gatekeeper subagent while preserving the existing workflow.

## Architecture

The first implementation is a workflow/agent contract change, not a config-system change. `samsara-bootstrap` asks for execution mode before `research`; if auto is selected, each later workflow question or confirmation dispatches `samsara-auto-gatekeeper`, appends a decision to `changes/<feature>/auto-decisions.md`, and proceeds only according to that recorded decision.

## Tech Stack

Markdown-based Samsara skills and agents, existing Python converter/validator tests, pytest.

## Key Decisions

- Session ask first: avoids over-engineering `samsara_config.yaml` before the core auto loop works.
- Reusable gatekeeper agent: matches "agent as human, human as agent" and avoids duplicating gate logic in every skill.
- No human fallback in first cut: uncertainty is recorded in `auto-decisions.md`, and security/privacy unknowns become high-uncertainty `reject` decisions rather than user questions.
- Append-only decision file: gives the auto run an auditable trace for every former human question or confirmation, including later corrections as new entries.
- Release bump included: after the feature is complete, release metadata moves from `0.10.0` to `0.11.0` using the existing marketplace source-of-truth workflow.

## Death Cases Summary

1. Auto mode silently skips a human gate instead of recording a gatekeeper decision.
2. `auto-decisions.md` exists but contains generic approvals that do not answer the stage question.
3. Auto mode secretly asks the user for confirmation after the run started.
4. Auto mode ships without a synchronized `0.11.0` release bump.
