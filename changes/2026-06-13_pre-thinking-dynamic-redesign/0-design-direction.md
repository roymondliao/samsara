## 1. Pre-thinking 要解決什麼問題

**它的工作**:在 planning 之前,把問題想清楚——這次到底要解什麼、系統怎麼設計、
做到哪、邊界在哪、東西放哪。

**為什麼需要它**:planning 想得好不好,取決於它拿到的資訊夠不夠。pre-thinking 就是
替 planning 把資訊備齊、把該對齊的決定先對齊。

**它真正在防的事**:系統實作完沒多久就開始爛、壞了沒人發現,最後新需求進不來、
只能整個砍掉重寫——這是 AI 寫程式最常見的結局。好系統是一路 iterate 上去的
(10% → 30% → 60% → 90%),pre-thinking 要保護的就是「之後還改得動」這件事。

**一個重要分寸**:「之後改得動」不等於「現在先把擴充功能蓋好」。會不會去改、往哪改,
要等上線後的回饋、數據、實際問題才知道,而那些現在都還不存在。所以 pre-thinking
不猜未來長什麼樣,只做一件事:**現在別把以後想改的路堵死**。比喻:別把門焊死,
但也別憑空蓋一條沒人走的走廊。

**這次重設計的主要目的是 auto mode**:讓 agent 能自己把設計想清楚,不必一直回頭
靠真人互動才知道怎麼做。所以整套設計的成功標準是:**在不出錯的前提下,盡量少需要
真人**——而不是「問出漂亮的問題」。

## 2. 這些決定的證據來自哪

方向不是憑感覺定的,是翻了三個真實 repo 的歷史紀錄(changes/)歸納出來的:

| Repo | 性質 | 看到了什麼 |
|---|---|---|
| samsara | 工具/skill | pre-thinking 常 1-2 輪就結束;有一次把四個提案綁成一句「要不要全做?」的引導式問題;連風險最高的改動也只問 2 輪 |
| xnexus-infra-engineer | infra / 雲端 | 品質明顯較高(infra 出錯代價大,人被逼著想仔細);但仍有兩個設計在實作中途被推翻(資料格式、版本命名),都是該問而沒問的;「填佔位值結果系統看似活著其實是壞的」出現三次 |
| vicone-ai-secondsight | 產品功能 | 早期文件很有深度(放哪、用什麼模式、要不要重用都想到了),但全是自言自語、沒人挑戰;而且**533 行的深度文件,10 天內萎縮到 16 行**——沒有結構強迫,深度約一週就退化成走過場 |

**歸納出三條鐵律**(整個設計建在這上面):

1. **問題不是 LLM 想不到**——它寫得出好東西。問題是「沒人要求它想、想完沒人挑戰」。
2. **沒有結構性壓力的深度,約一週就腐化成儀式。**
3. **不同類型的工作要想的東西大半不一樣**——所以「按類型切換思考」是必要的,不是花俏。

## 3. 六步流程

```
1. 定位這次的工作    ── 判斷類型 + 用同一把尺決定要想多深(全自動)
2. 立假設、框範圍    ── 把 research 結論變成明確假設,框出思考範圍
3. 多角度蒐證       ── 派獨立的角度去補盲點,只蒐集事實、不給建議
4. 收斂成設計決定    ── 一個 agent 一條龍,把每個決定鎖進三個格子
5. 把該問的問出來    ── 真選擇題問出來、記成痕跡(問真人或 gatekeeper)
6. 誠實交棒        ── 定一把驗收尺、做明確承諾,交給 planning
```

### 第一步:定位這次的工作

agent 自己從 research 結論 + 看一眼 codebase 判出三件事,寫下來,之後可被推翻:

**(1) 類型**(可同時是好幾種):feature / 效能優化 / infra / 重構 / 接外部服務 /
改資料結構。分不清就全都算上。
→ 為什麼:不同類型要想的東西不同(第四步據此決定想哪些)。

**(2) 深度**:用**一把尺**量——**不確定性 × 爆炸半徑**。
- 不確定性 = 有沒有「還沒想清楚、得靠猜」的設計問題。
- 爆炸半徑 = 萬一做錯,會擴散多遠?知不知道擴散到哪。
- 這把尺有三段,但**深想是預設守門員,fast-track 和輕想是要「證明」才能走的窄路徑**:
  - **fast-track**:證明兩軸都趨近零(沒有沒想清楚的問題、錯了損害有限且已知)→ 完全
    不用想。能跳過的根本原因不是「改動小」,「小」只是表面。
  - **輕想**:證明 main agent 自己看得全(沒有它想不到的盲點)→ main agent 自己蒐證,
    **不派平行搜尋者**。
  - **深想**:其他全部。**派平行搜尋者多角度蒐證**(第三步那套)。
- 輕想/深想的差別,實質就是**派不派平行搜尋者**;判準是「main agent 會不會有盲點」——
  爆炸半徑大(不知道擴散到哪)或不確定性高,就有盲點 → 深想。
- 規則單向:可以往深走,不准往淺走;要證明才能往淺(prove, not assume),不確定就算
  深想。判錯成深想只多花 token,判錯成輕想/fast-track 會漏掉盲點——代價不對稱。

**(3) 成熟度**(看**這次改動的範圍**,不是整個專案)——決定第四步「自己推導」能拿
什麼當根源:
- **全新 repo**:沒有舊程式可參考,推導只能靠 Samsara 公理 + 問題本質。
- **成熟 repo 裡加新元件**:可參考 repo 既有慣例(但引用前要先確認慣例本身沒爛)+ 公理。
- **成熟 repo 裡改既有模組**:可參考該模組契約 + 慣例(同樣先確認沒爛)+ 公理。
- 例:xnexus 的 github-mcp 是成熟 repo 裡的新元件,就合理地拿既有 hermes-agent 慣例
  當依據。

### 第二步:立假設、框範圍

把 research 的結論,收成一組**明確寫出來的假設**。這步是**收斂**(畫一個盒子,讓後面
的思考在盒子裡進行,不會越想越散),不是發散。

一條假設值不值得寫,用尺的「爆炸半徑」那一軸篩:**「如果這條是錯的,設計會不會跟著
改?」會改才寫,不會改是雜訊(留給實作)。** 立幾條由這把篩子決定,沒有固定數;不確定
要不要寫時就寫(寫一條很便宜,漏一個錯前提很貴)。

每條假設寫四欄:
```
假設:這次假定什麼成立
邊界:什麼情況成立;什麼情況不成立
萬一不成立:什麼會壞、誰最先發現
依據:有沒有可查證的證據可指
```

**「有把握 / 沒把握」用客觀準則判,不靠感覺**:
- **有把握 = 當場指得出一個具體、可查證的依據**(file:line / 既有測試 / research 寫下
  的發現 / 既有契約)。宣稱有把握就必須把依據寫出來,沒寫自動算沒把握。
- 不算依據的:「通常都這樣」「框架應該會處理」「業界慣例如此」——那是感覺,不是這個
  案子的證據。
- 這符合公理:存在即責任,有存在就會留下證據,拿得出證據才算扛起責任。

假設立完自然分兩種:**有把握的** → 框範圍用;**沒把握的** → 進第三步去蒐證(蒐不到、
只有真人知道的,變成之後要問的問題)。若假設彼此矛盾、兜不攏 → 代表 research 框錯了,
**退回 research**。

### 第三步:多角度蒐證

針對沒把握的假設去找證據。**為什麼要多角度、而不是 main agent 自己找**:因為 main
agent 只會去它「已經想到」的地方找,證實不了它沒料到的問題——它的盲點決定它找得到
什麼。派**不同角度、各自獨立的搜尋者**,才會撞到它沒想到要找的東西。這是多角度存在的
唯一理由:補盲點。

- **角度數量沒有上限**,這次需要幾個就派幾個,由證據「會散在哪幾類地方」決定。能沒
  上限是因為**搜尋者只帶事實回來,事實彼此不打架、只會互補**(若讓它們各自下判斷,
  多了就會收到一堆基於不同前提的矛盾建議——所以不讓它們下判斷)。
- **每個搜尋者帶回**:查到的事實(附出處)、沿路發現但原本沒列到的東西(這是補盲點的
  真正價值)、查不到的東西。**沒有「建議」這一欄。**
- **怎麼交派**:角度(要回答的問題)+ 思考範圍(第二步框的盒子)+ 起點(幾個入口檔案,
  但只是起點不是「只准查這些」,否則等於把 main agent 的盲點塞進搜尋者)+ 回傳格式。
- 某個角度失敗或回空白 → 那塊記成「未查證的缺口」,不准當沒事繼續。
- **跟深度連動**:輕想時假設少、證據近,main agent 自己看就好;深想才真的派獨立搜尋者
  多角度平行。
- 另有一份「常用角度清單」當提醒(不是上限):推完角度對一下有沒有漏掉已知重要的,
  刻意不用要寫理由。
- **codebase-map 當起點,不當真值**:有 `.samsara/codebase-map.yaml` 且夠新時,搜尋者
  拿它當起點假設(省得重挖),但**以 live codebase 為準**——map 跟現實不符時信現實,
  並把落差(drift)標出來。map 缺或過時 → 不准憑印象編,該查就查。(第一步判成熟度時
  也可先看 map。)
- **唯一寫入者**:搜尋者只把結果**回傳給 main agent**,自己不留檔;pre-thinking 檔
  由 main agent 一個人寫。這樣避免多個搜尋者同時寫同一份檔造成混亂。

### 第四步:收斂成設計決定

main agent **一個人、按順序**把證據收成設計決定(不能平行,因為決定彼此相依:「放哪」
要先知道「核心是什麼」)。這是最危險的一步——下判斷正是 LLM 最容易打回「看起來不錯就
這樣」的地方。

**先決定「這次要做哪些設計決定」(維度從哪來)**:不是套一份固定清單,而是**自然長
出來 + 用類型檢查表當安全網**(跟第三步角度同一套管法):
1. 從第二步**沒把握的假設**——每個「錯了設計會變」的假設,就是一個要做的決定。
2. 從第三步**證據與候選缺口**——搜尋者撞到的東西,變成要決定的維度。
3. 對照下面的**類型檢查表**——漏了已知重要的就補;真要跳過,寫一行理由(加維度
   免費、漏掉已知維度要理由)。

類型檢查表(提醒用,非上限;由第一步的類型決定看哪幾列):

| 類型 | 常見判斷維度 |
|---|---|
| feature | 放哪+理由 / 重用既有 or 新做 / 跟既有寫法一致 or 偏離(偏離寫成本) / 要繼承哪些上游契約 |
| 效能優化 | 現況基準+目標 / 怎麼量(→ 直接變第六步驗收尺) / 怎麼防退步 |
| infra | 格式契約(資料/secret/設定) / blast radius(含權限) / 佔位值生命週期(誰填、何時換、填錯有無警示) / rollback / 哪些驗證 agent 做、哪些 operator 做 |
| 重構 | 怎麼證明行為沒被改壞 / 安全的切割順序(分步、隨時可停) |
| 接外部服務 | 自建 or 用現成 / 退出成本 / 對方介面契約 |
| 改資料結構 | 相容性 / 新舊並存期 / 回退點 |

(這份維度表與第三步的角度表**不同層但對應**:角度蒐集事實,維度消費那些事實做
決定。例:結構角度蒐到的事實 → 餵「放哪 / reuse / pattern」維度。)

**每個維度,再用三個格子下判斷。核心機制一句話:每個決定都必須落進三個格子之一,
沒有第四個叫「我覺得」的格子。**

1. **證據已決定** —— 第三步的事實已逼出唯一答案,引用出處。
2. **自己推導** —— 證據沒逼出來,但能從一個**講得出名字的根源**推出來,而且**把推導鏈
   寫出來**。根源 = Samsara 公理某條 / 問題本身的硬需求 / 既有且確認沒爛的慣例。
   - 為什麼要寫出鏈、而不是只說「我用第一性原理想過」:後者可以對任何結論貼標籤、極易
     造假;寫出鏈,假的就露餡(接不回根源)。
   - 成熟 repo 的陷阱:既有程式有一部分是爛的。「跟既有一致」可能在沿用爛東西——引用
     既有慣例前先確認它沒爛,確認是爛的就偏離它、並把偏離記成一個決定。
3. **需要外部拍板** —— 推不出來,有好幾個答案都站得住腳,選哪個需要 agent 無權自己
   發明的偏好/優先序。答題者:**human,或 auto 模式的 gatekeeper**。
   - 但要再分兩種:(a) 能從專案原則/慣例推出的 → gatekeeper 名正言順地判;(b) 只有
     真人/組織才知道的(命名慣例、業務優先序)→ gatekeeper 只能猜,**必須標記「未經
     確認的猜測」並記錄**,讓它變成看得見的風險,而不是靜默的錯誤假設。

**可逆性折進這三格**:做每個決定時多問一句「以後要改有多痛」——
- 好改、或兩種選法一樣好改 → 自己選彈性那個(屬「自己推導」),順手記反轉成本。
- 想保留彈性得現在多花成本 → 沒有標準答案的取捨,屬「需要外部拍板」。
- 真的去蓋擴充點 → 只有當下已有實證才做,否則只記反轉成本、不動工。
- 特別標「看似好改、實際語義被鎖死」的偽可逆——它給虛假安全感,最先爛。

**收尾檢查**:只有「上面長出來的每個維度(含檢查表沒被寫理由跳過的)都落進了三格
之一」時,才能宣告「沒有缺口」。不准用一句「沒什麼好問的」帶過。

### 第五步:把該問的問出來

第四步認出並標記了哪些決定「需要外部拍板」;這步真的去問、並記下答案。最危險的地方
是:**問法本身會決定你拿到真決定還是橡皮圖章。** 所以兩條:

- **每個問題都是真選擇題**:至少兩個選項 + 各自取捨。絕不問「要不要照我推薦的做?」
  這種是非題(這正是 samsara 那個「要不要全做?」的毛病)。
- **答案記成痕跡**,不是只記結論:選了哪個 / 為何不選別的 / 這決定假設了什麼 / 那個
  假設垮掉時什麼最先爛。這樣後人才接得下去改,不必把整件事重吵一遍。

一道安全網:多角度合併時,**最刺耳、只有一個角度提出的候選最容易被靜默丟掉**——被丟
的要列出來寫明理由,沒丟寫「無」。

實務:human 用選擇題問;auto 由 gatekeeper 逐題答((b) 型標記為猜測)。一次問一小批,
題多分輪。答案若推翻了第四步某個推導,要往下游連帶修正(決定彼此相依)。

### 第六步:誠實交棒

交給 planning 前,留下兩樣,而且對自己的狀態誠實:

**(1) Evaluation Contract(唯一標準,沿用既有格式)**:定一個 Primary evaluator——什麼
東西成立就代表做完了,要 agent 能實際跑、實際查,不是「感覺做完」。**這不是另立的新
東西,就是現有的 Evaluation Contract**(Primary evaluator + pass / fail / feedback /
out-of-scope),格式沿用。
- 只能一個:iterate 一路往上爬,每個增量各換一把尺就分不清是進步還是漂移;下游
  planning / 實作 / debug / 驗收全指向同一個 Evaluation Contract。
- 在這裡定:設計意圖還完整在眼前、又還沒寫 code 的最後一刻,最準。

**(2) 一個明確承諾(三選一)+ 殘留清單**:**繼續 / 帶著已知缺口繼續 / 退回 research**。
- 逼出明確表態,就不可能一邊裝作全解決、一邊偷溜進 planning。
- 殘留清單 = 這裡解不了、要移交 planning 或之後處理的事(例如「這步要 operator 手動
  填值」)。
- auto:承諾一樣由 gatekeeper 做;「退回 research」允許(是流程轉向,不是退回找真人)。

## 4. 貫穿全程的設計原則

1. **凡是會被「感覺」蒙混的判斷,一律換成「指得出可查證的證據」**:把握、做完、推導,
   全是這個手法。LLM 可以假裝有把握,假裝不出一個指得到的 file:line。
2. **不對稱摩擦(整個設計的支點)**:讓「少想」處處有阻力、「多想」處處免費。想深不用
   理由,想淺要寫理由;加一個角度免費,拿掉已知角度要理由;挑戰假設免費,自我接受被
   禁止;記反轉成本便宜,憑空蓋擴充點昂貴。
3. **事實平行、判斷序列**:蒐集事實可多路平行(事實不打架);下判斷必須一條龍(判斷
   相依,平行會分岔出矛盾)。
4. **留下收斂痕跡**:每個決定帶「為何不選別的」和「哪裡最先爛」,給未來 iterate 的人
   當路標。
5. **寧可要會發警報的失敗**:優先選「壞了看得出來」的方案,而不是「壞了悄無聲息」的。
   猜測不可怕,靜默的猜測才可怕。

---

## 5. 怎麼知道這套機制壞了(上線後定期看)

任一條長期成立,代表對應機制已死:
- 「自己推導」裡偏離既有慣例的比例長期是零 → 沒在真檢查慣例有沒有爛,只在追認現狀。
- 反轉成本全標「容易改」→ 沒認真估(真實系統不可能每個決定都好改)。
- 幾乎不會走深想 → 深度門檻設太高。
- 「為何不選別的」開始出現跨案複製貼上的罐頭句 → 收斂痕跡在腐化。
- gatekeeper 的 (b) 型猜測標記長期是零 → 它在掩飾猜測,不是真的都推得出來。
- 某角度連續多案查不出東西 → 該淘汰;新角度從來沒人用 → 學習機制死了。
- pre-thinking 文件長度中位數掉回 50 行以下 → 深度整體退化(對照 secondsight 的
  533→16 曲線)。
- 搜尋者回傳夾帶設計建議 → 判斷被單一角度帶偏(回傳格式上就不該有建議欄)。

## 7. 否決了什麼(別再走回頭路)

- **第三格叫「隨便都行」**:否決。它是逃生門,會讓人把不想深想的都丟進去。改成「自己
  推導 + 寫出推導鏈」。
- **用「會不會 iterate」當閘門**:否決。連真人都不知道答案,不可回答的問題不該設成
  閘門。改成「反轉成本永遠評,只有『保留彈性要多花錢』的具體決定才問」。
- **憑「打算 iterate」就預留擴充點**:否決。憑空蓋的擴充點沒人用就是爛代碼,違反公理。
  改成「有當下實證才真的蓋」。
- **限制角度數量**:否決。搜尋者只取證、事實不打架,沒必要設上限。
- **把第二步當「發散亂想」**:否決。無邊界發散收不回來。改成「先框範圍」。
- **搜尋者也下判斷**:否決。判斷相依,平行會收到矛盾建議。限制成只取證。
- **深度判定與 fast-track 各用一把尺**:否決。同一個「簡單度」判兩次會打架。改成同一
  把尺(不確定性 × 爆炸半徑)三段:fast-track / 輕想 / 深想。

## 8. 還沒決定的

**已解:**
- **類型 → 判斷維度**:「自然長出 + 類型檢查表當安全網 + 漏掉要理由」,維度內容已
  補回第四步;可逆性移出類型、改為每個決定都問。
- **第一步「成熟度」判定**:已補回第一步,接上第四步「自己推導」的根源選擇。
- **codebase-map 的角色**:已補回第三步(當起點假設、live 為準、drift 要標)。
- **research 交什麼給第二步**:不變,就是 pre-thinking 現在已讀的 research 產出
  (kickoff、problem-autopsy)。
- **第六步驗收尺**:就是現有的 Evaluation Contract(唯一標準),沿用既有格式,不另立。
- **唯一寫入者**:搜尋者只回傳給 main agent、自己不留檔;main agent 是唯一寫入者。
- **第一步「可被推翻」**:就是既有的「深度單向閥 + 中途可升級」——後面步驟發現新證據,
  main agent 自己往上升(不准往下降)。非新機制。
- **第 7 項:輕想 / 深想的具體分界**:差別就是**派不派平行搜尋者**;判準是「main agent
  會不會有盲點」(爆炸半徑大或不確定性高 → 有盲點 → 深想)。fast-track 已定(兩軸歸零),
  通往輕想只有一條窄路徑(證明 main agent 看得全),其餘全部由深想守門。已寫進第一步。
